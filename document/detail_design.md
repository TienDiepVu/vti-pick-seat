## 2. Đặc tả Cơ sở dữ liệu Supabase (Database Schemas)

Ứng dụng sử dụng 2 bảng chính trong PostgreSQL để lưu giữ trạng thái hệ thống:

### 2.1. Bảng danh sách đăng ký ban đầu (`registrations`)
Lưu trữ thông tin của các CBNV đã hoàn tất thủ tục đăng ký tham gia sự kiện trước đó, và ghi nhận thêm các đăng ký phát sinh sau khi chương trình diễn ra.

*   **Đặc tả các cột (Columns Schema):**
    *   `account` (text, Primary Key): Tài khoản viết tắt của CBNV (ví dụ: `minh.phamthihong`).
    *   `employee_name` (text, Not Null): Họ và tên của CBNV đăng ký.
    *   `unit` (text, Not Null): Đơn vị làm việc (ví dụ: `VTI.D2`).
    *   `relatives` (text[], Default: '{}'): Mảng chứa danh sách họ tên của người thân đi cùng.
    *   `is_extra` (boolean, Default: false): Cờ đánh dấu tài khoản đăng ký phát sinh sau (được tự động thêm khi CBNV chưa đăng ký gốc nhưng đồng ý đặt ghế).

*   **SQL Khởi tạo:**
    ```sql
    CREATE TABLE public.registrations (
        account text NOT NULL PRIMARY KEY,
        employee_name text NOT NULL,
        unit text NOT NULL,
        relatives text[] DEFAULT '{}'::text[],
        is_extra boolean DEFAULT false
    );
    ```

### 2.2. Bảng lịch sử đặt ghế (`bookings`)
Lưu trữ toàn bộ thông tin các vị trí ghế đã được giữ chỗ thành công theo thời gian thực.

*   **Đặc tả các cột (Columns Schema):**
    *   `id` (bigint, Primary Key, Auto-increment): Định danh duy nhất cho từng lượt đặt ghế.
    *   `seat` (text, Not Null): Mã số ghế (ví dụ: `C3`).
    *   `name` (text, Not Null): Tên người tham gia ngồi ghế đó (CBNV hoặc tên người thân).
    *   `account` (text, Not Null): Tài khoản CBNV đứng tên đăng ký.
    *   `unit` (text, Not Null): Đơn vị của CBNV.
    *   `cinema` (text, Not Null): Rạp chiếu (`cinema-1` hoặc `cinema-2`).
    *   `saved_at` (timestamptz, Default: now()): Thời điểm đặt ghế thành công.
    *   *Ràng buộc duy nhất (Unique Constraint)*: `bookings_cinema_seat_key UNIQUE (cinema, seat)` để tránh 2 người cùng đặt 1 ghế tại một rạp chiếu.

*   **SQL Khởi tạo:**
    ```sql
    CREATE TABLE public.bookings (
        id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
        seat text NOT NULL,
        name text NOT NULL,
        account text NOT NULL,
        unit text NOT NULL,
        cinema text NOT NULL,
        saved_at timestamp with time zone DEFAULT now() NOT NULL,
        CONSTRAINT bookings_cinema_seat_key UNIQUE (cinema, seat)
    );
    ```

## 3. Thiết kế API Endpoints (API Specification)

Server Node.js tiếp nhận các request thông qua module HTTP và xử lý bất đồng bộ thông qua các endpoint sau:

### 3.1. Lấy danh sách ghế đã được đặt
*   **Endpoint:** `/api/bookings`
*   **Method:** `GET`
*   **Query Parameters:**
    *   `cinema` (String): Tên rạp cần lấy thông tin (`cinema-1` hoặc `cinema-2`).
*   **Phản hồi thành công (200 OK):**
    ```json
    {
      "seats": ["C3", "C4"]
    }
    ```

### 3.2. Đặt ghế & Xác thực thông tin
*   **Endpoint:** `/api/validate-booking`
*   **Method:** `POST`
*   **Request Payload (JSON):**
    ```json
    {
      "cinema": "cinema-1",
      "account": "minh.phamthihong",
      "unit": "VTI.D2",
      "attendees": [
        {
          "seat": "C3",
          "name": "Chu Tường Vy"
        }
      ],
      "allowUnregistered": false
    }
    ```
*   **Phản hồi thành công (Lưu thành công - 200 OK):**
    ```json
    {
      "valid": true,
      "seats": ["C3"]
    }
    ```
*   **Phản hồi lỗi nghiệp vụ (200 OK nhưng `valid: false`):**
    *   *Trường hợp CBNV chưa đăng ký chương trình:*
        ```json
        {
          "valid": false,
          "needsConfirm": true,
          "message": "Tài khoản này chưa đăng ký tham gia chương trình."
        }
        ```
    *   *Trường hợp vượt quá số lượng đăng ký tối đa:*
        ```json
        {
          "valid": false,
          "message": "Tài khoản này đã đăng ký X người, thêm Y người sẽ vượt quá số lượng ban đầu (Z người)."
        }
        ```
    *   *Trường hợp ghế đã được đặt (đè lên nhau):*
        ```json
        {
          "valid": false,
          "message": "Ghế C3 đã được đăng ký."
        }
        ```
*   **Phản hồi lỗi máy chủ (500 Internal Server Error):**
    ```json
    {
      "valid": false,
      "message": "Không kiểm tra được thông tin đăng ký."
    }
    ```

### 3.3. Lấy dữ liệu đối soát sự kiện
*   **Endpoint:** `/api/monitoring`
*   **Method:** `GET`
*   **Phản hồi thành công (200 OK):**
    Trả về danh sách 2 bảng thông tin đối soát được ánh xạ giữa bảng `registrations` và `bookings`:
    ```json
    {
      "mappedRows": [
        {
          "stt": 1,
          "participantName": "Chu Tường Vy",
          "employeeName": "Phạm Thị Hồng Minh",
          "account": "minh.phamthihong",
          "unit": "VTI.D2",
          "seat": "C3",
          "cinema": "cinema-1"
        }
      ],
      "unmappedRows": [
        {
          "stt": 1,
          "participantName": "Nguyễn Văn A",
          "employeeName": "",
          "account": "nguyen.vana",
          "unit": "VTI.D1",
          "seat": "A5",
          "cinema": "cinema-1",
          "note": "Không có trong danh sách đăng ký ban đầu"
        }
      ]
    }
    ```

### 3.4. Giữ chỗ ghế tạm thời (Locking seats)
*   **Endpoint:** `/api/hold-seats`
*   **Method:** `POST`
*   **Request Payload (JSON):**
    ```json
    {
      "cinema": "cinema-1",
      "seats": ["C3"],
      "sessionId": "session_abc123"
    }
    ```
*   **Phản hồi thành công (200 OK):**
    ```json
    {
      "valid": true
    }
    ```
*   **Phản hồi khi có tranh chấp (409 Conflict):**
    ```json
    {
      "valid": false,
      "message": "Ghế C3 đang được người khác chọn, vui lòng chọn ghế khác."
    }
    ```

### 3.5. Giải phóng ghế giữ chỗ tạm thời
*   **Endpoint:** `/api/release-seats`
*   **Method:** `POST`
*   **Request Payload (JSON):**
    ```json
    {
      "cinema": "cinema-1",
      "seats": ["C3"],
      "sessionId": "session_abc123"
    }
    ```
*   **Phản hồi thành công (200 OK):**
    ```json
    {
      "valid": true
    }
    ```

### 3.6. Xóa ghế đã chọn (Delete booking)
*   **Endpoint:** `/api/delete-booking`
*   **Method:** `POST`
*   **Request Payload (JSON):**
    ```json
    {
      "cinema": "cinema-1",
      "seat": "C3"
    }
    ```
*   **Phản hồi thành công (200 OK):**
    ```json
    {
      "success": true
    }
    ```
*   **Phản hồi lỗi (400 hoặc 500):**
    ```json
    {
      "success": false,
      "message": "Không thể xóa ghế đã đặt."
    }
    ```

## 4. Xử lý Logic Backend (`server.js`)

### 4.1. Chuẩn hóa chuỗi dữ liệu (`normalizeText`)
Nhằm loại bỏ sự khác biệt về chữ hoa/thường, khoảng trắng thừa hoặc lỗi gõ phím khi đối chiếu chuỗi:
```javascript
function normalizeText(value) {
  return String(value || "").trim().replace(/\s+/g, " ").toLowerCase();
}
```

### 4.2. Routing tĩnh và bảo mật (`serveStatic`)
*   Máy chủ tự phân tích đường dẫn URL để phục vụ các file HTML, CSS, JS trong thư mục `public`.
*   **Bảo mật:** Ngăn cản truy cập bất hợp pháp các file cấu hình bằng cách kiểm tra:
    ```javascript
    if (!filePath.startsWith(publicDir) || filePath.endsWith(".local.json")) {
      res.writeHead(403);
      res.end("Forbidden");
      return;
    }
    ```

### 4.3. Logic Đối soát (`getMonitoring`)
1.  Truy vấn bất đồng bộ đồng thời (`Promise.all`) toàn bộ bảng `registrations` và `bookings` từ Supabase.
2.  Chuyển đổi dữ liệu `registrations` thu được thành một Object để dễ dàng tra cứu theo `account`.
3.  Tạo một bản đồ ánh xạ nhanh (`bookedByPerson`) từ các ghế đã đặt, khóa ánh xạ là sự kết hợp: `normalizeText(Account) + "::" + normalizeText(Họ tên người tham gia)`.
4.  **Duyệt qua danh sách Đăng ký gốc**:
    *   Lấy ra CBNV và toàn bộ người thân đi kèm.
    *   Tìm kiếm trong bản đồ `bookedByPerson` xem họ đã chọn ghế chưa.
    *   Đẩy thông tin vào danh sách `mappedRows` (bao gồm cả trường `isExtra` để nhận diện đăng ký phát sinh). Nếu đã chọn ghế, ghi nhận mã ghế & rạp. Nếu chưa chọn, mã ghế & rạp để trống.
5.  **Duyệt qua danh sách Ghế đã đặt thực tế**:
    *   Tìm xem có lượt đặt nào có khóa `Account::Tên người tham gia` không nằm trong danh sách đăng ký gốc hay không.
    *   Nếu không tìm thấy, đẩy đối tượng này vào nhóm `unmappedRows` để làm bằng chứng đối soát sai lệch thông tin.

### 4.4. Logic Đặt và Kiểm duyệt ghế (`validateBooking`)
1.  Truy vấn thông tin đăng ký của `account` trên bảng `registrations` để kiểm tra sự tồn tại.
2.  Nếu không tồn tại:
    *   Nếu cờ `allowUnregistered` là `true`: Chèn bản ghi đăng ký mới vào bảng `registrations` với `is_extra = true`, đồng thời gán dữ liệu này làm `registration` để tiếp tục tiến trình.
    *   Nếu cờ `allowUnregistered` là `false`: Từ chối đăng ký và gửi mã phản hồi đề xuất xác nhận `needsConfirm: true`.
3.  Tính toán số lượng ghế đặt tối đa cho phép (`relatives.length + 1`).
4.  Đếm số lượng ghế đã đặt của `account` trên bảng `bookings` (`existingCount`).
4.  Kiểm tra nếu `existingCount + attendees.length > allowedCount` thì từ chối.
5.  Kiểm tra trùng lặp ghế tại rạp đã chọn.
6.  Chèn các dòng mới (`insert`) vào bảng `bookings`. Nếu gặp lỗi trùng lặp do xung đột ghi đồng thời (Postgres trả về lỗi code `23505`), gửi thông báo lỗi thân thiện cho client.
7.  **Giải phóng giữ chỗ tạm thời**: Khi lưu vào Supabase thành công, tự động xóa danh sách ghế đó khỏi cache `temporaryHolds` để mở khóa hoàn toàn.

### 4.5. Quản lý giữ chỗ tạm thời và Cơ chế Timeout
*   **Cơ chế lưu trữ**: Sử dụng đối tượng `temporaryHolds` lưu giữ trạng thái trong RAM của máy chủ Node.js để có phản hồi tức thời.
*   **Giải phóng tự động**: Một bộ lặp `setInterval` chạy mỗi 10 giây sẽ tự động kiểm tra và giải phóng các ghế có thời gian giữ quá **3 phút (180.000 ms)** mà chưa hoàn tất lưu trữ Supabase.
*   **Logic API `/api/hold-seats`**:
    *   Nếu ghế đã được lưu chính thức trong Postgres $\rightarrow$ Trả về lỗi `409`.
    *   Nếu ghế đang bị giữ bởi một `sessionId` khác trên RAM và chưa hết hạn $\rightarrow$ Trả về lỗi `409`.
    *   Ngược lại $\rightarrow$ Cập nhật thông tin giữ chỗ tạm thời và trả về `200`.

### 4.6. Logic xóa ghế đã đặt (`deleteBooking`)
1. Nhận thông tin rạp (`cinema`) và ghế (`seat`) cần xóa từ client.
2. Kiểm tra tính hợp lệ của tham số. Nếu thiếu, trả về lỗi `400`.
3. Kiểm tra xem ghế đó có thuộc hàng ghế đôi (bắt đầu bằng chữ `M`) hay không. Nếu là ghế đôi, tự động tính toán mã ghế cùng cặp tương ứng (ví dụ: lẻ $X$ đi kèm chẵn $X+1$, chẵn $X$ đi kèm lẻ $X-1$) để đưa cả hai ghế vào danh sách cần xóa.
4. Thực hiện lệnh xóa `DELETE` các bản ghi tương ứng trong bảng `bookings` của Supabase bằng bộ lọc `eq("cinema", cinema).in("seat", seatsToDelete)`.
5. Trả về kết quả JSON báo thành công hoặc thất bại.

## 5. Xử lý Logic Frontend

### 5.1. Trang chọn ghế (`public/js/seating.js`)

#### A. Cấu hình sơ đồ phòng chiếu (`configs`)
Từng hàng ghế được cấu hình số lượng và trạng thái đặc biệt:
*   `selected`: Khóa mặc định (Ví dụ hàng A, B của Cinema 1 & 2 đã bị khóa do mục đích kỹ thuật / BTC đặt trước).
*   `couple`: Ghế đôi nằm ở hàng M cuối cùng.
```javascript
const configs = {
  "cinema-1": [
    { row: "A", count: 12, state: "selected" },
    //...
    { row: "M", count: 10, state: "couple" }
  ]
}
```

#### B. Cơ chế chọn ghế đôi (`getSeatGroup` & `setSeatGroupCurrent`)
Khi bấm vào một ghế thuộc loại `couple`, hệ thống tự động tìm ghế đi kèm dựa vào thuộc tính `data-pair` (được tính bằng cách làm tròn `Math.ceil(vị trí ghế / 2)`).
Cả hai ghế trong cặp sẽ cùng thay đổi trạng thái chọn (`current`) đồng thời.

#### C. Responsive Sơ đồ ghế (`setScale`)
Để sơ đồ ghế luôn hiển thị trọn vẹn và không bị vỡ bố cục trên các màn hình khác nhau, hệ thống sử dụng thuật toán tính toán tỷ lệ scale động dựa trên kích thước cửa sổ so với khung hình chuẩn 1920x1080:
```javascript
function setScale() {
  const stage = document.querySelector(".stage");
  if (!stage) return;
  const scale = Math.min(stage.clientWidth / 1920, stage.clientHeight / 1080);
  stage.style.setProperty("--scale", scale);
}
```
Tại CSS, khung hiển thị sẽ biến đổi tương ứng: `transform: scale(var(--scale))`.

### 5.2. Trang đối soát (`public/js/monitoring.js`)

#### A. Tìm kiếm Realtime (`filterRows`)
Khi người dùng nhập từ khóa tìm kiếm, frontend sẽ tự động gộp tất cả các trường dữ liệu của một hàng thành một chuỗi duy nhất đã được chuẩn hóa thông qua hàm `normalizeText`. Sau đó thực hiện tìm kiếm chuỗi con bằng phương thức `.includes(keyword)`.
```javascript
const haystack = [
  row.stt,
  row.participantName,
  row.employeeName,
  row.account,
  row.unit,
  row.seat,
  row.cinema
].map(normalizeText).join(" ");
```
Phương pháp này giúp tìm kiếm đa tiêu chí nhanh chóng mà không cần gọi lại API server.

#### B. Logic Xóa ghế đã đặt
1. Mỗi hàng dữ liệu trên bảng đối soát sẽ hiển thị nút "Xóa" màu đỏ tương ứng nếu có ghế đã đặt.
2. Khi quản trị viên bấm vào nút "Xóa", hộp thoại xác nhận `window.confirm` xuất hiện để tránh thao tác nhầm lẫn.
3. Nếu đồng ý, frontend gửi yêu cầu `POST` tới API `/api/delete-booking` mang theo thông tin rạp và ghế.
4. Khi nhận phản hồi thành công từ backend, frontend sẽ gọi hàm `loadMonitoring()` để tải lại danh sách đối soát mới nhất, đồng thời cập nhật ngay lập tức giao diện.
